# Obsidian Knowledge Graph & Auto-Doc Skills Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build two Claude Code skills that generate and maintain an interconnected Obsidian knowledge graph documenting every feature, DB entity, UI component, and user flow in WhatStage.

**Architecture:** Two skills share template reference files. `obsidian-bootstrap` scans the codebase once to seed ~85 notes. `feature-doc` generates notes incrementally per new feature and auto-updates MOC indexes. All notes use Obsidian frontmatter and wikilinks for full graph connectivity.

**Tech Stack:** Claude Code skills (SKILL.md + references/), Obsidian Markdown with frontmatter + wikilinks, Mermaid diagrams

# Note Templates

These templates are used by both obsidian-bootstrap and feature-doc skills to generate consistent Obsidian notes.

## Feature Note Template

When creating a feature note, save to `whatstage_obsidian/Features/{Feature Name}.md`:

\```
---
tags: [feature, {subsystem-slug}]
status: planned
subsystem: {subsystem-slug}
created: {YYYY-MM-DD}
---

# {Feature Name}

## Description
{1-2 sentences: what this feature does and why it exists}

## Entities
- [[{table_name}]] — {role in this feature}

## Components
- [[{ComponentName}]] — {role}

## Flows
- [[{Flow Name}]]

## API Routes
- `{METHOD} {path}` — {purpose}

## Notes
_No additional notes yet._
\```

### Rules
- Feature names use Title Case with spaces: "Lead Pipeline", "Qualification Engine"
- File name matches the heading exactly: `Features/Lead Pipeline.md`
- Every entity, component, and flow reference MUST be a wikilink `[[Name]]`
- If no API routes exist yet, omit the section entirely
- Status values: `planned`, `in-progress`, `complete`

---

## Entity Note Template

When creating an entity note, save to `whatstage_obsidian/Entities/{table_name}.md`:

\```
---
tags: [entity, {subsystem-slug}]
table: {table_name}
subsystem: {subsystem-slug}
created: {YYYY-MM-DD}
---

# {table_name}

## Schema
| Column | Type | Constraints |
|--------|------|------------|
| {col} | {type} | {constraints} |

## Relationships
- Belongs to [[{parent_table}]] ({fk_column})
- Has many [[{child_table}]]

## Used By Features
- [[{Feature Name}]]

## Used By Components
- [[{ComponentName}]]

## RLS Policy
`{policy_name}` — {description}
\```

### Rules
- Entity names use the exact SQL table name in lowercase: `leads`, `workflow_steps`
- File name matches the table name: `Entities/leads.md`
- Parse schema from `supabase/migrations/*.sql` and `src/types/database.ts`
- Every FK column becomes a "Belongs to [[target_table]]" relationship
- Every table that FKs to this table becomes a "Has many [[source_table]]"
- If no RLS policy exists, write: "No RLS policy defined yet."

---

## Component Note Template

When creating a component note, save to `whatstage_obsidian/Components/{ComponentName}.md`:

\```
---
tags: [component, {type}, {subsystem-slug}]
file_path: {src/path/to/file.tsx}
route: {/route/path}
subsystem: {subsystem-slug}
created: {YYYY-MM-DD}
---

# {ComponentName}

## Description
{1 sentence: what this component renders or does}

## Route
`{/route/path}` ({context})

## Data Consumed
- [[{table_name}]] — {how it uses this data}

## Part Of
- [[{Feature Name}]]

## Source
`{src/path/to/file.tsx}`
\```

### Rules
- Component names use PascalCase: `LeadsPage`, `DashboardNav`
- The `type` tag is one of: `page`, `component`, `api-route`, `layout`, `middleware`
- The `context` in Route is one of: `marketing`, `tenant-dashboard`, `action-page`, `api`
- For API routes, the "Route" section shows the HTTP methods supported
- Derive the component name from the file path:
  - `src/app/(tenant)/app/leads/page.tsx` → `LeadsPage`
  - `src/app/api/fb/webhook/route.ts` → `FbWebhookRoute`
  - `src/components/dashboard/DashboardNav.tsx` → `DashboardNav`
  - `src/middleware.ts` → `Middleware`

---

## Flow Note Template

When creating a flow note, save to `whatstage_obsidian/Flows/{Flow Name}.md`:

\```

tags: [flow, {subsystem-slug}]
subsystem: {subsystem-slug}
created: {YYYY-MM-DD}
---

# {Flow Name}

## Diagram
```mermaid
flowchart LR
    A[{Step 1 label}] --> B[{Step 2 label}]
    B --> C[{Step 3 label}]
```

## Steps
1. **{Step description}** → [[{Feature or Entity or Component}]] {detail}
2. **{Step description}** → [[{Feature or Entity or Component}]] {detail}

## Entities Involved
- [[{table_name}]], [[{table_name}]]

## Components Involved
- [[{ComponentName}]], [[{ComponentName}]]
\```

### Rules
- Flow names use Title Case with spaces: "Lead Qualification Flow"
- Every step MUST have at least one wikilink to a Feature, Entity, or Component
- Mermaid diagram uses `flowchart LR` (left-to-right) for linear flows, `flowchart TB` for branching flows
- Keep diagrams to 5-10 nodes max — summarize groups of steps as single nodes if needed
- The "Entities Involved" and "Components Involved" sections list ALL wikilinks from the steps in one place for quick reference
```

```markdown
---
name: obsidian-bootstrap
description: One-time scan of the WhatStage codebase to seed the Obsidian knowledge graph vault with all entities, components, features, flows, and MOC index files
---

# Obsidian Bootstrap

Scan the existing WhatStage codebase and generate a fully interlinked Obsidian knowledge graph in `whatstage_obsidian/`.

## When to Use

- First time setting up the Obsidian vault
- After major codebase restructuring to resync the vault from scratch
- WARNING: Re-running deletes all content in `whatstage_obsidian/` (except `.obsidian/` settings) and regenerates from current codebase state

## Prerequisites

- The `whatstage_obsidian/` directory must exist at the project root
- Database schema must exist in `supabase/migrations/*.sql`
- TypeScript types must exist in `src/types/database.ts`

## Process

Follow these steps IN ORDER. Do not skip steps. Do not parallelize steps — each step depends on the previous one.

### Step 1: Clean the vault

Delete all content in `whatstage_obsidian/` EXCEPT the `.obsidian/` directory (which contains user settings). Then create the folder structure:

```
whatstage_obsidian/

├── Entities/
├── Components/
├── Flows/

### Step 2: Generate Entity notes

Read the database schema from `supabase/migrations/*.sql` and `src/types/database.ts`.

For each table:
1. Read the `references/templates.md` for the Entity Note Template
2. Read `references/subsystems.md` to get the subsystem slug for this table
3. Extract columns, types, and constraints from the SQL migration
4. Identify FK relationships (both "belongs to" and "has many")
5. Generate the note using the template
6. Save to `whatstage_obsidian/Entities/{table_name}.md`

Generate notes for ALL tables — both the 16 existing tables from `0001_init.sql` and the 7 new tables defined in the design spec:
- `knowledge_docs`, `knowledge_chunks`, `bot_rules`, `conversation_corrections`
- `qualification_forms`, `qualification_responses`, `action_conditions`

For new tables not yet in migrations, generate Entity notes with the schema from the design spec (`docs/superpowers/specs/2026-04-18-obsidian-knowledge-graph-design.md` sections 7.8 and 7.9) and mark them with a note: "_Table not yet in migrations — schema based on design spec._"

### Step 3: Generate Component notes

Scan the source tree:
- `src/app/**/*.tsx` — pages and layouts
- `src/app/**/route.ts` — API routes
- `src/components/**/*.tsx` — shared components
- `src/middleware.ts` — middleware

For each file:
1. Read `references/templates.md` for the Component Note Template
2. Read `references/subsystems.md` to map the file path to a subsystem
3. Derive the component name from the file path (see naming rules in templates.md)
4. Derive the route path from the directory structure
5. Determine which entities this component will consume (from the feature map)
6. Generate the note
7. Save to `whatstage_obsidian/Components/{ComponentName}.md`

### Step 4: Generate Feature notes

Read `references/feature-map.md` for the complete list of 41 features.

For each feature:
1. Read `references/templates.md` for the Feature Note Template
2. Use the subsystem, description, entities, and components from the feature map
3. Generate the note with all wikilinks to entities and components created in Steps 2-3
4. Save to `whatstage_obsidian/Features/{Feature Name}.md`

### Step 5: Generate Flow notes

1. Read `references/templates.md` for the Flow Note Template
2. Write a Mermaid `flowchart LR` (or `TB` for branching) diagram with 5-10 nodes
3. Write numbered steps with wikilinks to Features, Entities, and Components at each step
4. List all involved Entities and Components

The 10 flows to generate:

**Lead Qualification Flow** (subsystem: goals)
Steps: Lead messages on Messenger → Bot evaluates intent via AI → Bot sends qualification form button → Lead opens form → Lead fills quiz → Answers scored → Conditions evaluated → Lead tagged + stage moved → Workflow triggered → Confirmation sent via Messenger

**Appointment Booking Flow** (subsystem: goals)
Steps: Lead messages → Bot promotes availability → Bot sends booking button → Lead opens calendar page → Lead selects slot → Appointment created → Confirmation sent → Appointment visible on dashboard

**Product Purchase Flow** (subsystem: goals)
Steps: Lead messages → Bot showcases products → Bot sends catalog button → Lead browses catalog → Lead adds to cart → Lead completes checkout → Order created → Lead moved to Customer stage → Confirmation sent

**Messenger Webhook Flow** (subsystem: messenger)
Steps: Facebook sends webhook → Signature validated → Event parsed → Tenant resolved from page_id → Lead created/updated → Conversation logged → Bot flow triggered → Response sent

**Tenant Onboarding Flow** (subsystem: auth)
Steps: User signs up → Email confirmed → Onboarding page → Enter business name + slug → Select business type → Select bot goal → Connect Facebook page → Tenant created → Default stages seeded → Redirect to dashboard

**Bot Conversation Flow** (subsystem: messenger)
Steps: Lead sends message → Webhook received → RAG retrieves relevant knowledge → Bot rules applied → AI generates response → Goal-appropriate action button attached → Message sent → Lead response logged

**Workflow Execution Flow** (subsystem: workflows)
Steps: Event occurs (form submit/purchase/stage change) → Workflow triggers evaluated → Matching workflow started → Steps executed in order → Conditions evaluated at branches → Actions performed (send message/tag/move stage) → Run logged

**Action Page Submission Flow** (subsystem: actions)
Steps: Lead clicks action button in Messenger → Opens tenant subdomain page with PSID → Page renders based on type → Lead interacts → Submission created → lead_event logged → Workflows triggered → Notification sent via Messenger

**RAG Knowledge Retrieval Flow** (subsystem: rag)
Steps: Tenant uploads document → Document chunked → Chunks embedded → Stored in vector column → Lead sends message → Query embedded → Similar chunks retrieved → Chunks injected into AI prompt → AI responds with context

**Conversation Review Flow** (subsystem: rag)
Steps: Tenant opens conversation review → Browses past conversations → Flags bad bot response → Writes correction → Correction saved → Optionally applied as bot rule → Rule appears in rules list → Bot uses rule in future conversations

- [ ] **Step 1: Create the feature-doc skill definition**

```markdown
---
name: feature-doc
description: Auto-generate Obsidian knowledge graph notes for new features — creates Feature, Entity, Component, and Flow notes with wikilinks and updates MOC index files
---

# Feature Doc

Generate Obsidian notes for a newly implemented feature and wire it into the existing knowledge graph.

## When to Use

- After implementing a new feature, API route, component, or utility
- After adding a new database migration
- After creating a new user flow
- When modifying existing features that need their Obsidian notes updated

## Prerequisites

- The Obsidian vault must already be bootstrapped (run `obsidian-bootstrap` first)
- The vault structure must exist: `whatstage_obsidian/{Features,Entities,Components,Flows,Index}/`

## Process

### Step 1: Gather information

Ask the user (or detect from recent changes):

1. **What was built?** — feature name and brief description
2. **Which subsystem?** — check `references/subsystems.md` for valid subsystem slugs
3. **New DB tables?** — any new tables added in migrations
4. **New pages/components?** — any new files in `src/app/` or `src/components/`
5. **New user flow?** — does this feature involve an end-to-end user journey?
6. **Related existing features?** — which existing Feature notes should link to this new work?

### Step 2: Generate new notes

Based on what was built, create the appropriate notes using templates from `references/templates.md`:

**If new feature:**
- Create `whatstage_obsidian/Features/{Feature Name}.md`
- Set status to `in-progress` or `complete` based on implementation state
- Link to all relevant entities, components, and flows

**If new DB table(s):**
- Read the migration SQL to extract schema
- Create `whatstage_obsidian/Entities/{table_name}.md` for each new table
- Include full schema, FK relationships as wikilinks, RLS policies

**If new page(s)/component(s):**
- Create `whatstage_obsidian/Components/{ComponentName}.md` for each
- Include file path, route, data consumed, parent feature

**If new user flow:**
- Create `whatstage_obsidian/Flows/{Flow Name}.md`
- Include Mermaid flowchart diagram (5-10 nodes)
- Include numbered steps with wikilinks at each step

### Step 3: Update existing notes

For each new note created, update related existing notes to add wikilinks back:

**New entity created:**
- Open each Feature note listed in "Used By Features" → add the entity to the "## Entities" section
- Open each Component note listed in "Used By Components" → add the entity to the "## Data Consumed" section

**New component created:**
- Open its parent Feature note(s) → add the component to the "## Components" section

**New feature created:**
- Open each Entity note it references → add the feature to "## Used By Features"
- Open each Component note it references → add the feature to "## Part Of"

**New flow created:**
- Open each Feature note involved → add the flow to "## Flows"

### Step 4: Update MOC index files

Append new entries to the appropriate MOC files:

**Feature Roadmap** (`whatstage_obsidian/Index/Feature Roadmap.md`):
- If new feature: insert a new row BEFORE the `<!-- AUTO-UPDATED -->` marker
- Format: `| [[Feature Name]] | {subsystem} | 🟡 | [[entity1]], [[entity2]] | [[Component1]] |`

**Component Registry** (`whatstage_obsidian/Index/Component Registry.md`):
- If new component: insert a new row in the appropriate section BEFORE the `<!-- AUTO-UPDATED -->` marker
- Format: `| [[ComponentName]] | \`{route}\` | \`{file_path}\` |`

**Database Schema Map** (`whatstage_obsidian/Index/Database Schema Map.md`):
- If new entity: add the entity wikilink under its subsystem heading
- Update the Mermaid ER diagram to include the new table and its relationships

**System Overview** (`whatstage_obsidian/Index/System Overview.md`):
- Only update if a new subsystem is added (unlikely for incremental features)
- Update the subsystem table status if any feature status changed

### Step 5: Validate

Check all wikilinks in new AND modified notes:
- Every `[[wikilink]]` must point to an existing `.md` file in the vault
- Report any broken links and fix them

### Step 6: Report

Print what was created/updated:
- New notes created: {list}
- Existing notes updated: {list}
- MOC files updated: {list}
- Broken wikilinks found: {count}

### Step 7: Update feature map reference

If a new feature was created that isn't in `references/feature-map.md` (i.e., it's a feature discovered during development, not in the original 41), append it to the feature map reference file at `.claude/skills/obsidian-bootstrap/references/feature-map.md` so future bootstrap runs include it.
```
