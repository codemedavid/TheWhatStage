# Chatbot Awareness Redesign — Design

**Date:** 2026-04-25
**Status:** Design approved, awaiting implementation plan
**Supersedes (partially):** 2026-04-24-campaign-builder-v2-design.md (campaign config shape)
**Related:** 2026-04-19-campaign-flow-builder-design.md, 2026-04-25-action-button-messenger-sending-design.md

## Problem

The chatbot drifts off-topic. The triggering case: a campaign for selling WhatStage itself was configured with a phase prompt like "find out if the lead is running ads", and the bot interpreted this as a directive to do deep ad-setup discovery (asking about lead-gen setup, ad types, audience targeting, etc.) rather than as a single yes/no qualification check.

Two root causes were identified during brainstorming:

1. **Phase prompts are task-shaped.** Tenants write phases as instructions ("find out if X", "ask about Y"), and the LLM obeys literally — turning a guidepost into a script.
2. **Phases are treated as a state machine.** The bot walks leads through phases sequentially regardless of where the lead actually is. A hot lead saying "I want to buy" still gets dragged through earlier phases. (Hormozi calls this "re-cooling a hot lead.")

A third concern motivated the broader redesign: the chatbot was doing too much. Going forward, **action pages do the heavy lifting** (qualifying, collecting, selling); the chatbot's job narrows to FAQ + intent labeling + objection handling + reinforcement + routing.

## Approach

Replace the current free-form phase model with a **fixed Hormozi-derived awareness ladder** (Unaware → Problem-aware → Solution-aware → Product-aware → Most-aware). Phases become **observation-only stored state**, never constraints. The LLM's job each turn is to *label* the lead's awareness level; a deterministic router decides whether to surface a CTA and which one. Tenant configuration shifts from per-phase prompts to **campaign-level structured fields** (offer brief, top objections, knowledge scope, funnel type) from which phase strategies are auto-generated by template.

This is a structural shift along three axes:

- **Phase rigidity → awareness lens.** Phases stop driving behavior. They describe lead state.
- **LLM-as-router → LLM-as-labeler + deterministic routing.** Action page selection becomes a pure function of (awareness, funnel, qualifier-state).
- **Tenant-authored prompts → templated prompts from structured config.** Tenants can no longer write task-shaped phase prompts because they don't write phase prompts at all.

## Goals

- Eliminate off-topic drift on the WhatStage selling campaign and equivalent cases.
- Stop hot leads from being dragged through unnecessary discovery.
- Constrain RAG retrieval to campaign-relevant knowledge only.
- Reduce tenant configuration surface area while increasing predictability.
- Preserve existing webhook flow, action button signed-URL flow, lead-knowledge extraction, idle-summary, human-handoff, and A-B campaign assignment.

## Non-Goals (v1)

- **New** custom phase mode (free-form per-campaign phases). The runtime path for `phase_mode='custom'` is preserved (existing migrated campaigns continue to run on the legacy code path), but no UI is exposed to create *new* custom-mode campaigns. The `awareness_ladder` mode is the only authorable option in v1.
- Multi-action funnels beyond `direct` and `qualify_first`. No demo→call→buy chains. No conditional routing on qualifier *answers* (only on qualifier *completion*).
- Auto-tagging of knowledge docs. Tenants tag manually.
- Per-phase tenant editing. All phase content auto-generated from campaign config.
- Hormozi value-stack / scarcity / guarantee as structured campaign fields. Tenants use `campaign_rules` (free-text) for these.
- Per-objection RAG injection (objection-specific chunk retrieval).

## Architecture Overview

Per-message pipeline:

```
1. Detect awareness level   [LLM]
2. Detect intent label      [LLM]
3. Detect objection         [LLM, matched against campaign.top_objections]
4. Build prompt (templated, scoped RAG)  [DETERMINISTIC]
5. Generate reply           [LLM]
6. Pick action page         [DETERMINISTIC: pickActionPage(awareness, funnel, qualifier_done, wants_cta)]
7. Persist observed state + send Messenger reply
```

Steps 1–3 + 5 happen in a single LLM call returning structured JSON. The LLM never picks the action page; it only signals readiness via `wants_cta_now`.

### Hormozi mapping

The design is informed by Alex Hormozi's published frameworks:

- **Awareness ladder** (Eugene Schwartz via *$100M Leads*): Unaware → Problem-aware → Solution-aware → Product-aware → Most-aware. Used directly as the phase taxonomy.
- **CLOSER** (Clarify, Label, Overview pain, Sell vacation, Explain objections, Reinforce): The chatbot focuses on **L + E + R**; **C, O, S** are partially offloaded to action pages (forms, sales pages).
- **Stop qualifying, start closing** rule: Most-aware leads bypass qualifier in `qualify_first` funnels.
- **Four root objections** (money, time, fit, trust): UI offers these as templates when authoring `top_objections`.

## Data Model

### `campaigns` — added columns

| Column | Type | Notes |
|---|---|---|
| `optimization_goal` | enum: `SELL` / `BOOK` / `QUALIFY_COLLECT` | drives action-page validation + analytics |
| `funnel_type` | enum: `direct` / `qualify_first` | default `direct` |
| `primary_action_page_id` | uuid → `action_pages` | required |
| `qualifier_action_page_id` | uuid → `action_pages`, nullable | required only when `funnel_type='qualify_first'` |
| `phase_mode` | enum: `awareness_ladder` / `custom` | default `awareness_ladder`; `custom` reserved (not implemented in UI) |
| `offer_brief` | JSONB `{ dream_outcome: string, core_pain: string, why_us: string }` | three short fields, replaces vague description for prompt purposes |
| `top_objections` | JSONB array `[{ objection: string, counter_frame: string }]` | 3–5 entries |

Existing `campaign_rules` (text[]), `name`, `description`, `campaign_plan` retained. `campaign_plan` becomes optional (used by the campaign-builder-v2 conversational workflow to populate the structured fields above).

### `campaign_phases` — repurposed

For `awareness_ladder` mode: **5 rows auto-seeded** per campaign on creation, one per ladder rung.

| Column | Notes |
|---|---|
| `awareness_level` | new enum: `UNAWARE` / `PROBLEM_AWARE` / `SOLUTION_AWARE` / `PRODUCT_AWARE` / `MOST_AWARE` |
| `name` | display label, auto-set ("Unaware", "Problem-aware", etc.) |
| `surface_cta` | bool — defaults: `UNAWARE`/`PROBLEM_AWARE` = false, others = true |
| `order_index` | 1–5, locked |

Existing fields (`system_prompt`, `goals`, `transition_hint`, `tone`, `max_messages`, `action_button_ids`) are **deprecated for `awareness_ladder` mode** — Layer D content is computed by the prompt builder from campaign config. Fields kept on the table for `custom` mode.

### `knowledge_docs` — RAG scoping

| Change | Notes |
|---|---|
| add `is_global_faq` boolean | default `false` |
| new junction table `campaign_knowledge_docs (campaign_id, doc_id)` | many-to-many |

RAG SQL filter:
```sql
WHERE is_global_faq = TRUE
   OR id IN (
     SELECT doc_id FROM campaign_knowledge_docs
     WHERE campaign_id = $current_campaign_id
   )
```

Untagged, non-FAQ docs are never pulled. This is the structural fix for retrieval drift.

### `conversations` / `conversation_phases` — observed state

Add to whichever table currently tracks per-conversation phase state:

| Column | Notes |
|---|---|
| `current_awareness_level` | enum, replaces hard phase pointer for awareness_ladder mode |
| `current_intent_label` | text — bot-generated label like "comparing", "stuck on price" |
| `qualifier_completed_at` | timestamp, nullable — funnel routing input |
| `detected_objections` | text[] — running list, dedup-merged each turn |

These fields are observation-only. No code branches on them as constraints.

## Per-Turn AI Flow

```
1. Load context  [DETERMINISTIC]
   - campaign + offer_brief + top_objections + campaign_rules
   - conversation history (last N msgs)
   - prior current_awareness_level, current_intent_label,
     qualifier_completed_at, detected_objections

2. Scoped RAG retrieval  [DETERMINISTIC]
   - SQL filter (above)
   - top 5 chunks merged into prompt
   - if top similarity < threshold (e.g. 0.5): return zero chunks

3. Build system prompt  [DETERMINISTIC, templated]
   Layers (in order):
     A. Identity + offer_brief
     B. campaign_rules (hard guardrails)
     C. top_objections (with verbatim counter_frames)
     D. Phase strategy block — auto-generated for the prior awareness rung
        (recognition cues, response strategy, do-nots).
        On the very first turn (no prior awareness stored), default to UNAWARE.
     E. Scoped RAG chunks
     F. Routing context — "the active action button is X, surface it
        when natural; do NOT invent other CTAs"
        (X is pre-computed by the deterministic router using prior awareness;
         finalised after step 5 if awareness changes.)
     G. Structured output schema (step 4)

4. LLM call → structured JSON  [LLM]
   {
     "reply": "...",
     "detected_awareness": "PROBLEM_AWARE" | ...,
     "intent_label": "comparing solutions",
     "detected_objection": "money" | "time" | "fit" | "trust" | null,
     "wants_cta_now": true | false,
     "qualifier_just_completed": true | false
   }

5. Deterministic router  [DETERMINISTIC]
   pickActionPage(detected_awareness, funnel_type, qualifier_completed_at, wants_cta_now)
     → { action_page_id | null, cta_text }

   Rules:
     - UNAWARE / PROBLEM_AWARE → null (never push CTA)
     - SOLUTION_AWARE / PRODUCT_AWARE:
         direct funnel → primary
         qualify_first + qualifier not done → qualifier
         qualify_first + qualifier done → primary
     - MOST_AWARE → primary (always; bypasses qualifier — Hormozi rule)
     - if wants_cta_now=false AND awareness ≤ PROBLEM_AWARE → no CTA
     - phase row's surface_cta=false → no CTA

   Note: step 3F builds prompt using PRIOR awareness; if step 5 picks a
   different page based on NEW awareness, the LLM's reply may already
   reference an old CTA. Mitigation: the LLM is instructed to keep CTA
   reference indirect ("if you're ready, the button below…") and the
   router's choice replaces any specific button reference at send time.
   For v1, prompt uses PRIOR awareness; reroute on awareness change is
   accepted as a minor UX gap (lead may need one more turn).

6. Send reply + persist  [DETERMINISTIC]
   - Messenger send: reply text + (if action_page chosen) signed action button
   - Update conversation: current_awareness_level, current_intent_label,
     merge detected_objection, set qualifier_completed_at on flag
   - Increment matched campaign_phases row's message count
   - Async: extract lead knowledge (existing flow, kept)
```

### Action page routing function

Pure function, fully testable in isolation:

```ts
function pickActionPage(
  awareness: AwarenessLevel,
  funnelType: 'direct' | 'qualify_first',
  qualifierCompletedAt: Date | null,
  wantsCtaNow: boolean,
  surfaceCta: boolean,           // from campaign_phases row
  primaryActionPageId: string,
  qualifierActionPageId: string | null,
): { actionPageId: string | null; ctaText: string } {
  if (!surfaceCta) return { actionPageId: null, ctaText: '' };
  if (awareness === 'MOST_AWARE') return loadPrimary();
  if (awareness === 'UNAWARE' || awareness === 'PROBLEM_AWARE') {
    return { actionPageId: null, ctaText: '' };
  }
  // SOLUTION_AWARE or PRODUCT_AWARE
  if (!wantsCtaNow) return { actionPageId: null, ctaText: '' };
  if (funnelType === 'qualify_first' && !qualifierCompletedAt) {
    return loadQualifier();
  }
  return loadPrimary();
}
```

40-case table-driven test covers every combination.

## Phase Strategy Templates (Layer D)

Templates are deterministic. `{offer.dream_outcome}`, `{offer.core_pain}`, `{offer.why_us}`, `{primary_action.title}` are interpolated from campaign config. `{campaign.campaign_rules}` joined as bullets and appended to global rules.

### UNAWARE
```
LEAD STATE: Doesn't yet know they have a problem you solve.
RECOGNIZE BY: vague curiosity, off-topic chatter, no pain articulated.
YOUR JOB: Plant the pain in one line. Reference {offer.core_pain}.
         Ask one question that surfaces whether they relate to it.
         Do NOT pitch. Do NOT mention the product yet.
         Do NOT surface any CTA.
TONE: warm, curious, not pushy.
```

### PROBLEM_AWARE
```
LEAD STATE: Knows the pain, doesn't know solutions exist.
RECOGNIZE BY: complaining about {offer.core_pain} or its symptoms.
YOUR JOB: Validate the pain in one sentence. Tease that there's a path
         to {offer.dream_outcome} without pitching.
         Ask one clarifying question MAX to confirm fit.
         Do NOT run discovery. Do NOT list features.
         Do NOT surface a CTA unless they explicitly ask "how" or "what is it".
TONE: empathetic, confident.
```

### SOLUTION_AWARE
```
LEAD STATE: Knows solutions exist, comparing categories.
RECOGNIZE BY: "what's the best way to...", "should I do X or Y", general research questions.
YOUR JOB: Differentiate {offer.why_us} in one line.
         If they ask a factual question, answer from the knowledge base — short.
         When natural, route to {primary_action.title}.
         Do NOT compare yourself to specific competitors unless asked.
         Do NOT dump features.
TONE: helpful expert, not salesy.
```

### PRODUCT_AWARE
```
LEAD STATE: Knows your product, weighing it. Often objecting or comparing.
RECOGNIZE BY: questions about price, features, terms; "vs <competitor>"; hesitation phrases.
YOUR JOB:
  - If a top objection is detected → use the matching counter_frame VERBATIM
    from campaign config. Do not improvise.
  - Otherwise reinforce {offer.why_us} and route to {primary_action.title}.
  - Always surface CTA when momentum allows.
DO NOT volunteer discounts. Do NOT negotiate against yourself.
TONE: confident, direct, value-anchored.
```

### MOST_AWARE
```
LEAD STATE: Ready to buy. Just needs the path.
RECOGNIZE BY: "how do I sign up", "send me the link", "I'm in", explicit yes signals.
YOUR JOB: Give the path. ONE message: brief confirmation + the action button.
         No questions. No pitch. No discovery.
         If qualifier_first funnel AND qualifier not done: still go straight
         to primary CTA — never re-cool a hot lead.
TONE: short, action-oriented.
```

### Cross-rung guardrails (always appended)
```
GLOBAL RULES (always apply):
- Reply in 1–3 sentences max unless the user asked a deep factual question.
- Never list more than ONE action button per message.
- Never invent action pages, prices, or guarantees not in your context.
- If the user asks a factual question NOT in the knowledge base, say so
  briefly and route them to the appropriate page or human if available.
- {campaign.campaign_rules joined as bullets}
```

## Tenant UI

### Campaign creation form (5 steps)

1. **Goal & funnel**: optimization_goal (radio), funnel_type (radio), primary_action_page_id (dropdown), qualifier_action_page_id (dropdown, conditional).
2. **Offer brief**: dream_outcome, core_pain, why_us (each ~1 sentence).
3. **Top objections**: repeating block, 3–5 entries; pre-filled templates for the four Hormozi categories (money, time, fit, trust).
4. **Knowledge**: multi-select tenant docs for this campaign; per-doc "global FAQ" toggle. Validation warning if zero docs selected.
5. **Campaign rules** (optional): free-text bullets.

**No phase editor in v1.** Optional read-only "Preview phase strategies" panel renders Layer D for each rung so tenants can see what the bot will say.

### Removed UI (for awareness_ladder mode)

- Per-phase prompt editor.
- Per-phase action button selector.
- Free-form `campaign_plan` JSON editor.

These remain for `custom` mode (existing migrated campaigns).

## Migration

One-shot script:

1. For each existing `campaigns` row:
   - Set `phase_mode = 'custom'`. Existing phase rows + behavior preserved.
   - `optimization_goal`, `offer_brief`, `top_objections`, `primary_action_page_id`, `funnel_type` left null/default. Tenant fills on next edit.
2. For each existing `knowledge_docs` row:
   - Set `is_global_faq = true`. Preserves current "bot can pull anything" behavior.
3. In-flight conversations: keep existing phase pointers; new fields default null.

**No tenant breakage.** Old campaigns keep working in `custom` mode. New campaigns default to `awareness_ladder`. A "convert to ladder" button on existing campaigns clears phase rows, re-seeds 5 awareness rows, and requires `offer_brief` + `top_objections` to be filled.

**Cutover dogfood**: rebuild the WhatStage selling campaign in `awareness_ladder` mode with explicit offer_brief + objections + scoped knowledge. Becomes the reference implementation.

## Telemetry

- Per-message: log `detected_awareness`, `intent_label`, `detected_objection`, `wants_cta_now`, `action_page_chosen`.
- Per-conversation: funnel of awareness levels reached, time-to-MOST_AWARE, qualifier completion rate.
- Per-campaign aggregate: drop-off by awareness rung (replaces per-phase drop-off).

## Testing

### Unit
- `pickActionPage` — exhaustive table (5 awareness × 2 funnel × 2 qualifier × 2 cta-want × 2 surface_cta = 80 cases).
- Layer D template generator — for each awareness rung, given fixed `offer_brief` + `top_objections`, snapshot rendered prompt block.
- RAG scoping SQL filter — fixture (campaign_id, doc tagging) → expected doc subset. Includes negative case: untagged non-FAQ docs excluded.
- Decision parser — bot output JSON missing fields, malformed JSON, unknown awareness enum → safe defaults (default to prior awareness, no CTA).
- Migration script — fixture: existing campaigns + phases + docs → expected new column values.

### Integration
- Webhook → conversation engine → reply, for each awareness rung: feed sample message that should trigger that rung, assert reply matches strategy expectations and `current_awareness_level` is stored.
- Hot-lead bypass: lead in `qualify_first` campaign sends "I want to buy" → router returns primary CTA, not qualifier.
- Objection routing: lead says "too expensive" → reply includes the campaign's money counter-frame text verbatim.
- Off-topic question: lead asks something not in scoped knowledge → bot says "I don't have that handy" + routes; does NOT pull global tenant docs.

### E2E (Playwright)
- Tenant creates a `qualify_first` campaign with offer_brief + 3 objections + scoped docs → publishes.
- Simulated Messenger conversation walks the awareness ladder; assert bot does NOT drill discovery on PROBLEM_AWARE.
- The exact bug scenario: WhatStage-selling campaign + lead says "I run ads" → bot does NOT ask about ad setup; instead labels intent and routes.

### Regression guard
- Snapshot existing prompt-builder output for `custom`-mode migrated campaigns — must not change after schema migration. Custom mode runs on the old code path.

## Open Questions

None blocking. The following are intentionally deferred to future iterations:

- Should `wants_cta_now=true` on `PROBLEM_AWARE` ever surface a CTA? (Currently no; Hormozi-strict.)
- Should we expose `intent_label` to tenant analytics in v1, or just log internally?
- For `qualify_first` funnels, is qualifier completion detected via webhook from the action page, or via LLM signal `qualifier_just_completed`? (v1: both — webhook is authoritative, LLM signal is a fallback for cases where the action page didn't fire a webhook.)
