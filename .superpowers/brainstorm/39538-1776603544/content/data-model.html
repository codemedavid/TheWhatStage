Design — Section 1: Data Model

New tables and how they replace / extend the existing schema

New Tables

campaigns

id, tenant_id
name, description
goal: 'form_submit' | 'appointment_booked' | 'purchase' | 'stage_reached'
goal_config: jsonb
is_primary: boolean
status: 'draft' | 'active' | 'paused' | 'archived'
follow_up_delay_minutes: int (default 120)
follow_up_message: text
created_at, updated_at

campaign_phases (replaces bot_flow_phases)

id, campaign_id, tenant_id
name, order_index
max_messages (soft hint)
system_prompt, tone, goals
transition_hint
action_button_ids: uuid[]
image_attachment_ids: uuid[]
created_at

lead_campaign_assignments

id
lead_id → leads (unique — one per lead)
campaign_id → campaigns
assigned_at

Permanent sticky assignment. Created on first message.

experiments

id, tenant_id
name
status: 'draft' | 'running' | 'paused' | 'completed'
started_at, ended_at
winner_campaign_id → campaigns
min_sample_size: int (default 50)
created_at

experiment_campaigns (join)

experiment_id → experiments
campaign_id → campaigns
weight: int (normalized %)
PK: (experiment_id, campaign_id)

campaign_conversions

id
campaign_id → campaigns
lead_id → leads
converted_at
metadata: jsonb (order_id, appt_id, etc.)

Old table kept temporarily, dropped after migration verified.

Assignment logic on first message

1. Lead sends first message → check lead_campaign_assignments
2. If none: check for a running experiment → pick campaign by weighted random → insert assignment
3. If no experiment: assign to the tenant's is_primary campaign → insert assignment
4. All subsequent messages → always load phases from assigned campaign
