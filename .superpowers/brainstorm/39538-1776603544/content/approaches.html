Campaign System — Architecture Approaches

Three ways to model campaigns, A/B testing, and phase metrics

A

Flat Campaigns + Implicit A/B

Each campaign has an is_primary flag and a test_weight. When multiple campaigns have in_test = true, the engine distributes new leads by weight. No explicit experiment entity.

Pros

  • Simplest schema
  • Fastest to build
  • Easy to reason about

Cons

  • No clear "start/end" of a test
  • Harder to archive old test results
  • Messy metrics over time
B

Campaigns + Experiments Recommended

Campaigns are standalone (one is primary). An experiment groups 2–4 campaigns together for a defined test period, with assignment weights and a conversion goal. The primary campaign runs outside of experiments.

Pros

  • Clean separation of prod vs test
  • Experiments have start/end & history
  • Winner suggestion tied to experiment
  • Natural "promote winner" flow

Cons

  • One extra table (experiments)
  • Slightly more UI to build
C

Campaign Groups (Hierarchical)

A campaign group is the top-level entity. Each group has one "control" (primary) and N "variants". Metrics always roll up to the group. A campaign can't exist outside a group.

Pros

  • Clean rollup analytics
  • Every campaign has context

Cons

  • Forces grouping even when not testing
  • More complex for "I just want one campaign"
  • Extra hierarchy adds cognitive overhead

Why B is recommended

A tenant usually has one good campaign running. Testing is a deliberate, time-bounded activity. Approach B models exactly that — normal operation stays simple, and when you want to test, you create an experiment. Old experiment results stay clean and queryable forever. It also maps directly to how real A/B testing tools (Optimizely, GrowthBook) work.
