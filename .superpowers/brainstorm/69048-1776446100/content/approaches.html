Obsidian Knowledge Graph + Auto-Doc Skill

Three approaches for building the system — pick the one that fits best

A

Single Monolithic Skill

One Claude Code skill (feature-doc) that handles everything: bootstrap scan, new feature generation, MOC updates, and flow diagrams — all in one SKILL.md file.

Pros

  • Simple — one skill to invoke
  • All logic in one place
  • Easy to maintain

Cons

  • Large skill file, harder to read
  • Can't run bootstrap vs incremental separately
  • Mixes concerns (scanning code vs generating docs)
B

Two Skills: Bootstrap + Feature-Doc

Split into obsidian-bootstrap (one-time scan of codebase → seed vault) and feature-doc (triggered per new feature → generates notes + updates MOCs). Shared templates in a references/ folder.

Pros

  • Clean separation of concerns
  • Bootstrap runs once, feature-doc runs often
  • Shared templates keep notes consistent
  • Each skill is focused and readable

Cons

  • Two skills to remember
  • Must keep templates in sync
C

Skill + Obsidian Plugin

Claude Code skill generates the notes, plus a custom Obsidian plugin that provides live views, auto-updates graph colors, and validates link integrity inside Obsidian itself.

Pros

  • Richest Obsidian experience
  • Live validation of broken links
  • Custom graph coloring by node type

Cons

  • Significant extra dev effort (Obsidian plugin API)
  • Plugin needs maintenance as Obsidian updates
  • Overkill for documentation — tags + folders achieve 90% of this

💡 Recommendation: Option B

Two focused skills with shared templates give us the best balance. Bootstrap handles the one-time heavy lift of scanning 16 DB tables, 30+ source files, and generating the initial graph. Feature-doc stays lean for daily use — just tell it what you built and it generates the right notes with all the wikilinks. Shared Markdown templates in references/ keep everything consistent.
