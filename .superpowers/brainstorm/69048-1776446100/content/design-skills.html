Design Section 2: The Two Skills

How obsidian-bootstrap and feature-doc work together

Skill 1: obsidian-bootstrap
PURPOSE

One-time scan of the existing codebase to seed the Obsidian vault with all current entities, components, features, and flows.

WHEN TO USE

Run once at project start, or re-run if you want to resync the vault from scratch.

WHAT IT DOES
  1. Scan DB schema — parse supabase/migrations/*.sql and src/types/database.ts → generate Entity notes for all 16 tables
  2. Scan source files — glob src/app/**/* and src/components/**/* → generate Component notes for every page and component
  3. Infer features — group related entities + components into Feature notes based on subsystem (leads, commerce, messenger, workflows, action pages, tenants)
  4. Generate initial flows — create core user journey flows based on CLAUDE.md descriptions
  5. Build MOCs — create all 4 Index files (System Overview, DB Schema Map, Feature Roadmap, Component Registry)
  6. Wire wikilinks — cross-reference everything: entities ↔ features ↔ components ↔ flows
OUTPUT

~40-50 Obsidian notes fully interlinked in the vault

Skill 2: feature-doc
PURPOSE

Triggered when implementing a new feature. Auto-generates Obsidian notes for all new pieces and updates existing notes + MOCs.

WHEN TO USE

After adding a new feature, API route, component, DB table, or user flow.

WHAT IT DOES
  1. Identify what's new — ask what was built (or detect from recent changes)
  2. Generate notes — create Feature, Entity, and/or Component notes using shared templates
  3. Generate flow — if it's a user-facing feature, create a Flow note with Mermaid diagram
  4. Update existing notes — add wikilinks to related existing entities/features/components
  5. Update MOCs — append new entries to Feature Roadmap, Component Registry, DB Schema Map
  6. Validate links — check that all [[wikilinks]] in new notes point to existing notes
OUTPUT

3-8 new/updated Obsidian notes per feature, all wikilinked
